# Supabase Schema Reference

This document describes the database tables used by the Orchestrator, their
field definitions, and the mapping rules applied when scrape output is
persisted.

---

## Table: `jobs_final`

Primary table for all job listings from ingest through enrichment.

| Column | Type | Nullable | Notes |
|--------|------|----------|-------|
| `id` | `uuid` | No | Primary key, generated by DB |
| `company_name` | `text` | Yes | |
| `role_title` | `text` | Yes | |
| `job_url` | `text` | Yes | Upsert conflict key |
| `description` | `text` | Yes | Markdown-formatted when scraped |
| `match_score` | `float` | Yes | Set by enricher |
| `saved_at` | `timestamptz` | Yes | ISO-8601 UTC |
| `job_status` | `text` | Yes | See status transition table below |
| `is_deleted` | `boolean` | No | Soft-delete flag |
| `modified_at` | `timestamptz` | Yes | |
| `language` | `text` | Yes | Default: `"English"` |
| `content_hash` | `text` | Yes | SHA-256 of description |
| `location` | `text` | Yes | |
| `source_platform` | `text` | Yes | Job-board identifier |
| `job_type` | `text` | Yes | See valid values below |
| `work_mode` | `text` | Yes | See valid values below |
| `tech_stack` | `text[]` | Yes | Set by enricher |
| `experience_level` | `text` | Yes | Set by enricher |
| `decision` | `text` | Yes | |
| `reason` | `text` | Yes | |
| `confidence` | `float` | Yes | |

### Status Transitions

```
SCRAPED  →  ENRICHED  →  (APPLIED | REJECTED | SAVED)
```

| Status | Set by |
|--------|--------|
| `SCRAPED` | Ingest stage (pipeline or submit endpoint) |
| `ENRICHED` | Enricher service |
| `APPLIED` | Manual / automation |
| `REJECTED` | Manual / automation |
| `SAVED` | Default for manually added rows |

### Valid `job_type` Values

`fulltime`, `parttime`, `internship`, `contract`

### Valid `work_mode` Values

`remote`, `hybrid`, `on-site`

## Scrape-Origin Field Mapping

When job rows are created by the scraping pipeline (i.e. via
`service/mappers/scrape_to_jobs_final.py`), the following field mapping is applied
before the `JobsFinalRow` schema validation gate.

| Scrape Output Field | `jobs_final` Column | Transformation |
|---------------------|---------------------|----------------|
| `company_name` | `company_name` | Pass through |
| `role_title` | `role_title` | Pass through |
| `job_url` | `job_url` | Stripped; raises `ValueError` if empty |
| `description` | `description` | Pass through |
| `job_type` | `job_type` | Pass through |
| `work_mode` | `work_mode` | Pass through |
| `location` | `location` | Pass through |
| `source_platform` | `source_platform` | Pass through |
| `language` | `language` | Defaults to `"English"` if absent |
| `content_hash` | `content_hash` | Pass through |
| `scraped_at` | `saved_at` | ISO-8601 UTC string from scraper |
| `id` *(scrape UUID)* | *(dropped)* | DB generates its own PK |
| `description_source` | *(dropped)* | Scraper metadata, not persisted |

Fields not present in scrape output that appear in `JobsFinalRow`
(`match_score`, `tech_stack`, `experience_level`, `decision`, `reason`,
`confidence`, `modified_at`) are left absent; the schema validator fills them
with `None`.
